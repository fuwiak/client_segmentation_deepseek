# Техническое задание: Интеллектуальная CRM для цветочного ритейла

> Статус документа: черновик v0.2. Фиксирует видение продукта и целевую
> архитектуру. Текущая реализация — MVP на XLSX-файлах (сегментация клиентов).
> Остальные модули заведены как абстракции/placeholder и включаются постепенно.

## 1. Цель системы

Создать **универсальную** (не заточенную под один магазин) CRM-платформу для
цветочного ритейла — переход от «хранилища контактов» к **интеллектуальному
хабу**, который сам обогащает данные. Система должна:

1. Собирать данные о клиентах из **разных источников** (Мой Склад, 1С,
   мессенджеры WhatsApp/Telegram/Instagram DM, Excel-выгрузки).
2. Складывать их в **единую нормализованную базу** с историей заказов,
   предпочтениями, средними чеками, получателями, поводами, важными датами.
3. Использовать **AI-агента** (кандидат — Hermes), который дополняет базу:
   анализирует переписки и прошлые заказы, извлекает факты, сегментирует,
   формирует рекомендации.
4. Выводить данные в **оболочку CRM** со статистикой, когортным анализом,
   воронкой лидов, единым окном чата с подсказками AI.
5. Интегрироваться с **входящими лидами** и формировать **рекламные кампании
   и таргетинг** с рекомендациями агента; на определённом этапе диалог
   передаётся живому оператору.
6. Повышать **LTV** (пожизненную ценность клиента) за счёт персонализации,
   рекуррентных напоминаний и событийного календаря.

**Главное отличие подхода:** CRM не привязана к конкретному магазину — задаётся
уникальная структура данных, наполняется из любых операционных систем магазина,
обогащается агентом и отображается в едином интерфейсе оператора.

## 2. Границы текущего этапа (MVP)

- Источник данных: **XLSX-файлы**, которые уже есть (`Контрагенты`, `Заказы`).
- Активный модуль: **AI-сегментация клиентов** (заполнение колонок
  `Группы`, `Заказчик или получатель`, `Пол`, `ТГ ник`).
- UI: **текущий** (FastAPI + HTMX), менять пока не требуется.
- Мой Склад, 1С, мессенджеры, лиды, кампании, Hermes — **placeholder/абстракции**,
  реализуются позже без переписывания ядра.

## 3. Архитектура (слои)

Ключевой принцип: **источники, хранилище, агенты и CRM-логика развязаны через
интерфейсы (ABC)**. Замена реализации не должна затрагивать остальные слои.

```
┌──────────────────────────────────────────────────────────────┐
│                         UI (FastAPI + HTMX)                    │  app/templates, app/main.py
└───────────────┬──────────────────────────────┬───────────────┘
                │                              │
        ┌───────▼────────┐            ┌────────▼─────────┐
        │   CRM-слой     │            │   Agents-слой    │  app/agents/
        │  app/crm/      │            │  сегментация,    │
        │  статистика,   │            │  обогащение,     │
        │  лиды, кампании│            │  кампании        │
        └───────┬────────┘            └────────┬─────────┘
                │                              │
        ┌───────▼──────────────────────────────▼───────┐
        │              Repository (хранилище)           │  app/repository/
        │        абстракция + in-memory (сейчас)        │
        │           SQL/Postgres (позже)                │
        └───────────────────────┬───────────────────────┘
                                │
        ┌───────────────────────▼───────────────────────┐
        │                 Domain-модели                  │  app/domain/
        │  Customer, Recipient, Order, PreferenceProfile,    │
        │  ImportantDate, Interaction, Lead, Campaign        │
        └───────────────────────▲───────────────────────┘
                                │
        ┌───────────────────────┴───────────────────────┐
        │                  Connectors                     │  app/connectors/
        │  Excel (сейчас) | МойСклад | 1С | Мессенджеры   │
        │        нормализуют источник → Domain            │
        └────────────────────────────────────────────────┘
```

### 3.1. Domain (`app/domain/`)
Единая модель данных — «истина» системы. Не зависит ни от источника, ни от БД.
Основные сущности: `Customer`, `Recipient`, `Order`, `PreferenceProfile`,
`ImportantDate`, `Interaction`, `Segment`, `Lead`, `Campaign`, `Recommendation`.

### 3.2. Connectors (`app/connectors/`)
Каждый источник реализует интерфейс `DataSourceConnector`:
`fetch_customers()`, `fetch_orders()`, `fetch_interactions()`. Возвращает
**Domain-объекты**, а не сырые строки. Сейчас реально работает `ExcelConnector`,
остальные — placeholder.

### 3.3. Repository (`app/repository/`)
Абстракция `CustomerRepository` (CRUD + запросы). Сейчас — `InMemoryRepository`,
позже — SQL/Postgres. UI и агенты работают только через интерфейс.

### 3.4. Agents (`app/agents/`)
Интерфейс `Agent` с методом `run(context) -> result`. Реализации:
- `SegmentationAgent` — активен (обёртка над текущей логикой).
- `EnrichmentAgent` — placeholder (анализ переписок → пол/имя/предпочтения).
- `CampaignAgent` — placeholder (таргетинг и рекомендации по кампаниям).

Внутренний движок агентов сейчас — OpenRouter (DeepSeek). Позже сюда можно
подключить Hermes-agent, реализовав тот же интерфейс `Agent`.

### 3.5. CRM (`app/crm/`)
Бизнес-логика поверх repository: `StatsService` (статистика — можно считать уже
сейчас), `LeadService` и `CampaignService` (placeholder).

## 4. Архитектура данных (структура БД)

База нормализована для масштабируемости, но гибка для AI-анализа. Разные
источники заполняют разные подмножества полей; AI-агент дозаполняет пробелы.

### 4.1. Основные сущности

| Сущность | Ключевые поля | Примечание |
|----------|---------------|------------|
| **Customer** (заказчик) | id, external_ids{источник→id}, ФИО, телефон, email, tg_ник, пол, дата рождения, адреса, сегменты[] (LTV, частота, лояльность), средний чек, всего заказов, дата последнего заказа, баллы, каналы[], источник, архивный, ai_summary | Кто **платит**. Сегменты — правила + AI |
| **Recipient** (получатель) | id, customer_id, имя, отношение (жена, мама, коллега…), пол, preference_profile_id, заметки, ai_labels | Кому **дарят**. Отдельная таблица — критично для цветочного ритейла |
| **Order** | id, customer_id, recipient_id, дата, сумма, маржинальность, статус оплаты/отгрузки, канал продаж, склад, позиции[] (цветы, упаковка), повод/тип (романтика, бизнес, соболезнование), адрес доставки, данные получателя (если ≠ заказчик), комментарии | Связь заказчик → получатель → состав букета |
| **PreferenceProfile** (AI-driven) | id, owner_type (customer/recipient), любимые цветы[], нелюбимые/аллергии[], стиль (минимализм, гигантизм…), ценовой диапазон, предпочтения доставки, confidence, source (agent/manual) | Профиль предпочтений — отдельно для заказчика и каждого получателя |
| **ImportantDate** | id, customer_id, recipient_id?, тип (ДР, годовщина, 8 марта…), дата (день/месяц, год опционален), recurrence (ежегодно), источник (agent/order/manual), confidence | Заполняется агентом из переписок: «У жены ДР 15 мая» |
| **Interaction** | id, customer_id, канал (tg/wa/instagram/звонок), направление, текст, дата, вложения, транскрипт (для звонков), метки AI | История переписки и звонков |
| **Segment** | код, название, правило/описание, источник (правило/AI) | VIP, постоянный, спящий и т.д. |
| **Lead** | id, источник, контакт, статус воронки, привязка к customer, назначенный оператор | Входящие обращения |
| **Campaign** | id, название, сегмент-таргет, канал, оффер, триггер (дата/событие/ручной), метрики, статус | Маркетинговые рассылки |
| **Recommendation** | id, customer_id/recipient_id/segment, тип, текст, автор (agent), confidence | Офферы и подсказки оператору |

### 4.2. Разделение заказчика и получателя

Критическая ошибка многих цветочных CRM — смешивание «кто платит» и «кому
дарят». Система **обязана** хранить:

- «Цветы для мамы — любит полевые»
- «Цветы для девушки — любит розы»

Это позволяет делать **два разных оффера одному клиенту** и строить
персонализацию по получателю, а не только по заказчику.

### 4.3. Доменные модели (`app/domain/models.py`)

Все сущности из п. 4.1 описаны в коде. `Customer`, `Order`, `Interaction`,
`Lead`, `Campaign`, `Recommendation` используются в MVP; `Recipient`,
`PreferenceProfile`, `ImportantDate` — целевая схема для Postgres-репозитория
(этап 2 дорожной карты).

## 5. Интеллектуальный слой: Hermes Agent

В качестве ядра предлагается модель семейства **Hermes** (OpenHermes на базе
Mistral или Llama 3) — хорошо справляется с извлечением сущностей (Entity
Extraction) и следованием инструкциям. Подключается через интерфейс `Agent`
(`app/agents/base.py`); сейчас движок — OpenRouter (DeepSeek), замена не ломает
вызывающий код.

### 5.1. Функции агента

| Функция | Описание | Статус |
|---------|----------|--------|
| **Обогащение (Enrichment)** | Парсит историю сообщений и заказов, извлекает факты: «У жены ДР 15 мая», «Не любит лилии», «Доставка до 10 утра» → заполняет `PreferenceProfile`, `ImportantDate`, `Recipient` | Placeholder (`EnrichmentAgent`) |
| **Сегментация** | Анализ стиля общения и готовности к тратам → VIP-группы, LTV, частота | Активен (`SegmentationAgent`) |
| **Рекомендации** | Перед праздниками генерирует оффер: «В прошлом году красные розы на 10 000 ₽, в этом году пионы — предложить?» | Placeholder (`CampaignAgent`) |
| **Подсказчик оператору** | При входящем лиде: «Это Иван, всегда берёт на 5к, предложи тюльпаны — спрашивал месяц назад» | Заготовка в UI чата |

### 5.2. RAG (Retrieval-Augmented Generation)

Чтобы агент не галлюцинировал, использовать **векторную БД** (Pinecone,
Weaviate, pgvector). В индексе:

- описания текущих букетов и композиций;
- наличие цветов на складе;
- актуальные цены и акции.

Агент сопоставляет желания клиента с **реальным ассортиментом**, а не выдумывает
позиции. RAG подключается на этапе 6–7 дорожной карты.

### 5.3. Гибридная модель общения (режим «Подсказчик»)

Не давать AI полную свободу переписки сразу:

1. Агент пишет **черновик ответа**.
2. Оператор нажимает «Отправить» или правит текст.
3. Правки сохраняются для обучения и снижения репутационных рисков.

Режим по умолчанию на проде; полная автономия — только после накопления
доверия и метрик качества.

### 5.4. Hermes vs GPT-4 / LiteLLM

| Критерий | Hermes (локально) | GPT-4 (API) |
|----------|-------------------|-------------|
| Парсинг больших историй | Дёшево, можно локально | Дорого при объёмах |
| Защита ПДн (152-ФЗ) | Данные не уходят наружу | Требует DPA |
| «Продающие» тексты | Средне | Отлично |

**Рекомендация:** гибрид через [LiteLLM](https://github.com/BerriAI/litellm) —
Hermes для enrichment/entity extraction, OpenAI для финальной генерации
кампанийных текстов.

### 6.2. Интерфейс оператора (Dashboard) — `app/crm/`, UI

- **Единое окно чата** с подсказками AI (гибридный режим).
- **Карточка клиента** с AI-сводкой (3 предложения по истории заказов).

- Метрики: open / click / conversion.

## 7. Сценарий работы (Workflow)

```
1. Сбор       Источники (1С, МойСклад, WhatsApp, Excel) → единая БД
2. Анализ     Hermes проходит историю → заполняет пустые поля карточек
3. Лид        Входящий запрос → агент идентифицирует клиента → подсказка оператору
4. Реактивация Событие в календаре → агент готовит текст → оператор «Отправить»
5. Кампания   Сегмент + триггер → AI-оффер → утверждение → рассылка
```

**Событийный календарь** — главная ценность в цветах (рекуррентность). CRM
становится «внешней памятью» клиента: напоминание о годовщине за неделю с
предложением прошлогоднего букета даёт конверсию, близкую к 90%.

## 8. Пайплайн данных

```
Источники → Connector.normalize() → Domain → Repository.upsert()
      → EnrichmentAgent (переписки/заказы) → Recipient, PreferenceProfile, ImportantDate
      → SegmentationAgent (сегменты) → Repository.update()
      → CRM: статистика / лиды / кампании → UI
      → CampaignAgent + RAG → рекомендации → оператор (human-in-the-loop)
```

## 9. Рекомендации и комментарии (от разработчика)

1. **Идентификация клиента (дедупликация)** — критическая проблема. В разных
   источниках один человек = разные записи (телефон, имя, tg). Нужен модуль
   matching/merge (по нормализованному телефону, email, tg). Заложить `external_ids`
   и ключ слияния сразу — переделывать потом дорого.

2. **Разделение заказчика и получателя** — отдельная таблица `Recipient` с
   собственным `PreferenceProfile`. Без этого персонализация в цветочном
   ритейле не работает.

3. **Нормализация телефонов** — привести всё к `+7XXXXXXXXXX` (`normalize_phone`
   в domain). Лучший стабильный ключ для склейки источников.

4. **Хранилище**: для прод-CRM нужен **Postgres** (связи, аналитика, поиск,
   pgvector для RAG). In-memory — только для MVP на XLSX.

5. **Персональные данные (152-ФЗ)**: телефоны, имена, переписки — ПДн. На проде
   нужны: согласие, шифрование чувствительных полей, ограничение доступа,
   логи. Hermes локально снижает риск утечки при парсинге историй.

6. **AI-агент и качество**: держать `confidence` и `reasoning` у каждого
   AI-заполнения. Human-in-the-loop: низкая уверенность → подтверждение
   оператором, не автозапись в источник.

7. **Идемпотентность и стоимость**: кэшировать результаты агента по хешу входных
   данных клиента, чтобы не гонять LLM повторно на неизменных записях.

8. **Мессенджеры**: начать с **экспортов переписок** (файлы), затем API
   (Green API, Telegram Bot). Самый ценный и самый сложный источник.

9. **RAG**: подключать до масштабирования рекомендаций — без каталога букетов
   агент будет предлагать несуществующие позиции.

10. **С чего начать разработку**: Backend-структура БД + скрипт-анализатор,
    который берёт выгрузку из мессенджера и с помощью Hermes составляет профили
    10–20 клиентов. Это покажет «чистоту» данных и ценность AI-аналитики до
    полноценного UI.

11. **Тестируемость**: каждый слой мокается через ABC. Connector — на
    XLSX-фикстурах, repository — in-memory, агент — на записанных ответах LLM.

## 10. Дорожная карта

- **Этап 1 (сейчас)**: XLSX → сегментация, текущий UI. ✅
- **Этап 2**: Postgres-репозиторий, `Recipient` / `PreferenceProfile` /
  `ImportantDate`, дедупликация, нормализация телефонов.
- **Этап 3**: EnrichmentAgent на экспортах переписок; скрипт профилирования
  10–20 клиентов; статистика в CRM.
- **Этап 4**: Мой Склад / 1С коннекторы (двусторонняя синхронизация).
- **Этап 5**: Лиды + воронка + гибридный чат (подсказчик) + передача оператору.
- **Этап 6**: Кампании + триггеры по `ImportantDate` + CampaignAgent.
- **Этап 7**: Hermes-agent + RAG (pgvector) + LiteLLM-роутинг моделей.

## 11. Структура интерфейса

### 11.1. Дашборд (`/dashboard`)
- Фильтр периода: день, неделя, месяц, год, за всё время, произвольный диапазон.
- KPI: всего клиентов, заказов, выручка — с % прироста к предыдущему периоду.
- Клиенты с маркетплейсов и разбивка по площадкам.
- Прямые продажи и разбивка по источникам.
- Повторные клиенты.
- ИИ-ассистент для вопросов по метрикам (заготовка).

### 11.2. AI база клиентов (`/clients`)
- Поля как в Excel + вычисляемые: `Тип продаж` (маркетплейс/прямые, ≠ канал продаж), `Статус последнего заказа`.
- Быстрый фильтр: маркетплейс / прямые / все.
- Раскрытие заказов клиента (второй уровень).
- AI-поля: группы, теги, пол, заказчик/получатель, ВИП, постоянный, саммари — с подсветкой и ref-источником.

### 11.3. Карточка клиента (модал)
- ФИО, роль, тип продаж, теги, контакты, средний чек, заказы, лояльность.
- Последний заказ + все заказы, рекомендация AI, быстрые кнопки (заготовка).

### 11.4. Главная (`/`)
- Открытые задачи, диалоги, статусы заказов.
- Список клиентов с фильтрами, ИИ-ассистент (заготовка).

### 11.5. Маркетинговые кампании (`/campaigns`)
- Создание кампании с фильтрацией по сегментам, канал WA/TG.

### 11.6. Настройки
- `/settings` — интеграции (Green API, Telegram, МойСклад).
- `/settings/communications` — автокоммуникация: принятие заказа, сбор, доставка, ОС, напоминание через 3 дня.

### 11.7. Интеграции мессенджеров
