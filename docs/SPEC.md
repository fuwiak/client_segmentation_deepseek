# Техническое задание: CRM для цветочного бизнеса с AI-агентом

> Статус документа: черновик v0.1. Фиксирует видение продукта и целевую
> архитектуру. Текущая реализация — MVP на XLSX-файлах (сегментация клиентов).
> Остальные модули заведены как абстракции/placeholder и включаются постепенно.

## 1. Цель

Создать **универсальную** (не заточенную под один магазин) CRM-систему для
цветочного бизнеса, которая:

1. Собирает данные о клиентах из **разных источников** (Мой Склад, 1С,
   мессенджеры, Excel-выгрузки).
2. Складывает их в **единую нормализованную базу** клиентов с историей заказов,
   предпочтениями, средними чеками, получателями, поводами и т.д.
3. Использует **AI-агента**, который дополняет базу: анализирует переписки и
   прошлые заказы, восстанавливает пол, имя получателя, сегменты, ТГ-ник и др.
4. Выводит данные в **оболочку CRM** со статистикой, воронкой лидов,
   отчётами.
5. Интегрируется с **входящими лидами** и формирует **рекламные кампании и
   таргетинг** с рекомендациями AI-агента; на определённом этапе диалог
   передаётся живому оператору.
6. Имеет **внутреннего агента** (кандидат — Hermes-agent) как ядро автоматизации.

## 2. Границы текущего этапа (MVP)

- Источник данных: **XLSX-файлы**, которые уже есть (`Контрагенты`, `Заказы`).
- Активный модуль: **AI-сегментация клиентов** (заполнение колонок
  `Группы`, `Заказчик или получатель`, `Пол`, `ТГ ник`).
- UI: **текущий** (FastAPI + HTMX), менять пока не требуется.
- Мой Склад, 1С, мессенджеры, лиды, кампании, Hermes — **placeholder/абстракции**,
  реализуются позже без переписывания ядра.

## 3. Архитектура (слои)

Ключевой принцип: **источники, хранилище, агенты и CRM-логика развязаны через
интерфейсы (ABC)**. Замена реализации не должна затрагивать остальные слои.

```
┌──────────────────────────────────────────────────────────────┐
│                         UI (FastAPI + HTMX)                    │  app/templates, app/main.py
└───────────────┬──────────────────────────────┬───────────────┘
                │                              │
        ┌───────▼────────┐            ┌────────▼─────────┐
        │   CRM-слой     │            │   Agents-слой    │  app/agents/
        │  app/crm/      │            │  сегментация,    │
        │  статистика,   │            │  обогащение,     │
        │  лиды, кампании│            │  кампании        │
        └───────┬────────┘            └────────┬─────────┘
                │                              │
        ┌───────▼──────────────────────────────▼───────┐
        │              Repository (хранилище)           │  app/repository/
        │        абстракция + in-memory (сейчас)        │
        │           SQL/Postgres (позже)                │
        └───────────────────────┬───────────────────────┘
                                │
        ┌───────────────────────▼───────────────────────┐
        │                 Domain-модели                  │  app/domain/
        │  Customer, Order, Interaction, Lead, Campaign  │
        └───────────────────────▲───────────────────────┘
                                │
        ┌───────────────────────┴───────────────────────┐
        │                  Connectors                     │  app/connectors/
        │  Excel (сейчас) | МойСклад | 1С | Мессенджеры   │
        │        нормализуют источник → Domain            │
        └────────────────────────────────────────────────┘
```

### 3.1. Domain (`app/domain/`)
Единая модель данных — «истина» системы. Не зависит ни от источника, ни от БД.
Основные сущности: `Customer`, `Order`, `Interaction`, `Segment`, `Lead`,
`Campaign`, `Recommendation`.

### 3.2. Connectors (`app/connectors/`)
Каждый источник реализует интерфейс `DataSourceConnector`:
`fetch_customers()`, `fetch_orders()`, `fetch_interactions()`. Возвращает
**Domain-объекты**, а не сырые строки. Сейчас реально работает `ExcelConnector`,
остальные — placeholder.

### 3.3. Repository (`app/repository/`)
Абстракция `CustomerRepository` (CRUD + запросы). Сейчас — `InMemoryRepository`,
позже — SQL/Postgres. UI и агенты работают только через интерфейс.

### 3.4. Agents (`app/agents/`)
Интерфейс `Agent` с методом `run(context) -> result`. Реализации:
- `SegmentationAgent` — активен (обёртка над текущей логикой).
- `EnrichmentAgent` — placeholder (анализ переписок → пол/имя/предпочтения).
- `CampaignAgent` — placeholder (таргетинг и рекомендации по кампаниям).

Внутренний движок агентов сейчас — OpenRouter (DeepSeek). Позже сюда можно
подключить Hermes-agent, реализовав тот же интерфейс `Agent`.

### 3.5. CRM (`app/crm/`)
Бизнес-логика поверх repository: `StatsService` (статистика — можно считать уже
сейчас), `LeadService` и `CampaignService` (placeholder).

## 4. Единая структура базы клиентов (черновик схемы)

| Сущность | Ключевые поля |
|----------|---------------|
| **Customer** | id, external_ids{источник→id}, имя, телефон, email, tg_ник, пол, дата рождения, адреса, сегменты[], средний чек, всего заказов, дата последнего заказа, баллы, предпочтения[], каналы[], источник, архивный |
| **Order** | id, customer_id, дата, сумма, статус оплаты/отгрузки, канал продаж, склад, позиции[], получатель, повод/событие, комментарии |
| **Interaction** | id, customer_id, канал (tg/wa/звонок), направление, текст, дата, вложения, метки AI |
| **Segment** | код, название, правило/описание, источник (правило/AI) |
| **Lead** | id, источник, контакт, статус воронки, привязка к customer, назначенный оператор |
| **Campaign** | id, название, сегмент-таргет, канал, оффер, метрики, статус |
| **Recommendation** | id, customer_id/segment, тип, текст, автор (agent), confidence |

Поля намеренно избыточны: разные источники заполняют разные подмножества,
AI-агент дозаполняет пробелы.

## 5. Пайплайн данных

```
Источники → Connector.normalize() → Domain → Repository.upsert()
      → EnrichmentAgent (переписки/заказы) → Repository.update()
      → SegmentationAgent (сегменты) → Repository.update()
      → CRM: статистика / лиды / кампании → UI
      → CampaignAgent → рекомендации → оператор
```

## 6. Рекомендации и комментарии (от разработчика)

1. **Идентификация клиента (дедупликация)** — критическая проблема. В разных
   источниках один человек = разные записи (телефон, имя, tg). Нужен модуль
   matching/merge (по нормализованному телефону, email, tg). Заложить `external_ids`
   и ключ слияния сразу — переделывать потом дорого.

2. **Нормализация телефонов** — привести всё к `+7XXXXXXXXXX`. Это лучший
   стабильный ключ для склейки источников. Сделать утилиту в domain.

3. **Хранилище**: для прод-CRM нужен **Postgres** (связи, аналитика, поиск).
   In-memory — только для MVP на XLSX. Абстракция `CustomerRepository` позволит
   переключиться без изменения агентов и UI. Рекомендую SQLModel/SQLAlchemy.

4. **Персональные данные (152-ФЗ)**: телефоны, имена, переписки — ПДн. На проде
   нужны: согласие, шифрование чувствительных полей, ограничение доступа,
   логи. Заложить это в модель хранения заранее.

5. **AI-агент и качество**: держать `confidence` и `reasoning` у каждого
   AI-заполнения (уже сделано). Для прод — human-in-the-loop: значения с низкой
   уверенностью отправлять на подтверждение оператору, не писать в источник
   автоматически.

6. **Идемпотентность и стоимость**: кэшировать результаты агента по хешу входных
   данных клиента, чтобы не гонять LLM повторно на неизменных записях.

7. **Мессенджеры**: анализ переписок — самый ценный, но и самый сложный
   источник (доступ к Telegram/WhatsApp, приватность). Начать с экспортов
   переписок (файлы), затем API. Абстракция `MessengerConnector` это учитывает.

8. **Кампании и таргетинг**: начать с сегмент-ориентированных рассылок
   (сегмент → оффер → канал), метрики open/click/conversion. AI даёт
   рекомендации оффера под сегмент, оператор утверждает отправку.

9. **Hermes-agent**: подключать как реализацию интерфейса `Agent`, чтобы ядро
   не зависело от конкретного движка. Пока — не трогаем.

10. **Тестируемость**: каждый слой мокается через свой ABC. Connector можно
    тестировать на XLSX-фикстурах, repository — на in-memory, агент — на
    записанных ответах LLM.

## 7. Дорожная карта (предложение)

- **Этап 1 (сейчас)**: XLSX → сегментация, текущий UI. ✅
- **Этап 2**: Postgres-репозиторий, дедупликация клиентов, нормализация телефонов.
- **Этап 3**: EnrichmentAgent на экспортах переписок; статистика в CRM.
- **Этап 4**: Мой Склад / 1С коннекторы (двусторонняя синхронизация).
- **Этап 5**: Лиды + воронка + передача оператору.
- **Этап 6**: Кампании + таргетинг + CampaignAgent.
- **Этап 7**: Hermes-agent как ядро автоматизации.

## 8. Структура интерфейса (по ТЗ заказчика)

### 8.1. Дашборд (`/dashboard`)
- Фильтр периода: день, неделя, месяц, год, за всё время, произвольный диапазон.
- KPI: всего клиентов, заказов, выручка — с % прироста к предыдущему периоду.
- Клиенты с маркетплейсов и разбивка по площадкам.
- Прямые продажи и разбивка по источникам.
- Повторные клиенты.
- ИИ-ассистент для вопросов по метрикам (заготовка).

### 8.2. AI база клиентов (`/clients`)
- Поля как в Excel + вычисляемые: `Тип продаж` (маркетплейс/прямые, ≠ канал продаж), `Статус последнего заказа`.
- Быстрый фильтр: маркетплейс / прямые / все.
- Раскрытие заказов клиента (второй уровень).
- AI-поля: группы, теги, пол, заказчик/получатель, ВИП, постоянный, саммари — с подсветкой и ref-источником.

### 8.3. Карточка клиента (модал)
- ФИО, роль, тип продаж, теги, контакты, средний чек, заказы, лояльность.
- Последний заказ + все заказы, рекомендация AI, быстрые кнопки (заготовка).

### 8.4. Главная (`/`)
- Открытые задачи, диалоги, статусы заказов.
- Список клиентов с фильтрами, ИИ-ассистент (заготовка).

### 8.5. Маркетинговые кампании (`/campaigns`)
- Создание кампании с фильтрацией по сегментам, канал WA/TG.

### 8.6. Настройки
- `/settings` — интеграции (Green API, Telegram, МойСклад).
- `/settings/communications` — автокоммуникация: принятие заказа, сбор, доставка, ОС, напоминание через 3 дня.

### 8.7. Интеграции мессенджеров
- **WhatsApp**: Green API (`GREEN_API_ID_INSTANCE`, `GREEN_API_TOKEN`).
- **Telegram**: Bot API (`TELEGRAM_BOT_TOKEN`).
- Переменные задаются через Railway CLI, не коммитятся в репозиторий.
