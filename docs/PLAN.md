# План разработки: Client Segmentation CRM

> Документ фиксирует, **что нужно сделать**, чтобы довести проект от текущего состояния до рабочей системы поиска «горячих кучек» клиентов и (в будущем) персонализированных коммуникаций.
>
> Прод: [client-segmentation-deepseek.up.railway.app/clients](https://client-segmentation-deepseek.up.railway.app/clients)

---

## Зачем это всё

**Конечная цель (фаза 3):** на основе данных о клиентах и заказах находить сегменты с высоким потенциалом и отправлять им релевантные предложения → рост конверсии и LTV.

**Пример цепочки:**
1. Фаза 2 отвечает: *«Антон и Ганди регулярно покупают цветы на ~1 млн ₽ за последние 2 месяца, пик — март, получатели — постоянные»*
2. Фаза 3 превращает это в кампанию: персональное предложение / напоминание / upsell
3. Конверсия растёт, потому что предложение попадает в реальную потребность, а не в «рассылку всем»

Фазы 1 и 2 — фундамент. Без полных и актуальных данных и без NL-аналитики фаза 3 бессмысленна.

| Область | Статус |

| Загрузка из **Excel** (.xlsx) | ✅ Работает |
| Страница `/clients` (таблица, фильтры, карточка) | ✅ Работает |

**Главная проблема сейчас:** данные на `/clients` могут приходить из МС или файлов, но пайплайн **не замкнут** — после синка нужно отдельно запускать AI, источники не сливаются, нет «спроси базу на человеческом языке».

## Фаза 1 — Единая база клиентов из МойСклад или файлов

### Цель

На странице `/clients` **все строки** — это реальные клиенты с заказами, загруженные **только** из:
- API **МойСклад** (основной источник на проде), или
- **Excel-выгрузок** (резерв / миграция / офлайн)

Никаких «пустых» или полусырых данных. Оператор открывает `/clients` и видит актуальную картину.

### Что уже есть

- Синк МойСклад → `DataHub` (контрагенты, заказы, позиции, кэш Redis)
- Парсер Excel + привязка заказов к контрагентам
- Автосинк при старте (`moysklad_auto_sync`)
- Вычисляемые поля: средний чек, дата последнего заказа, тип продаж, статус
- Экспорт обогащённой таблицы в XLSX

### Что нужно сделать

#### 1.1. Зафиксировать источник истины на проде

- [ ] На Railway: `MOYSKLAD_ENABLED=true`, валидный токен, `MOYSKLAD_AUTO_SYNC=true`
- [ ] Проверить, что после деплоя `/clients` заполняется без ручной загрузки Excel
- [ ] Добавить на `/clients` явный индикатор источника: **«МойСклад»** / **«Excel»** / **«нет данных»** + дата последнего синка

#### 1.2. Автоматический AI-пайплайн после загрузки данных

Сейчас AI-поля (`Группы`, `Саммари`, `Пол`…) требуют отдельного захода на `/segment` или кнопку «Обогатить». Для фазы 1 это должно быть **по умолчанию**.

- [ ] После успешного синка МойСклад → автозапуск сегментации (фон, с прогрессом на `/clients`)
- [ ] После загрузки Excel → тот же автозапуск
- [ ] Кнопка «Пересегментировать» на `/clients` (перезапуск без повторного синка)
- [ ] Статус в UI: «AI: готово / в процессе / ошибка / не запускалось»

#### 1.3. Довести полноту данных МойСклад

- [ ] Убедиться, что позиции заказов догружаются (фоновый `refresh_moysklad_positions`)
- [ ] Проверить маппинг всех нужных полей в `mapper.py` (телефон, канал продаж, баллы, адрес, теги МС)
- [ ] Обработка краевых случаев: контрагент без заказов, заказ без контрагента, дубликаты телефонов

#### 1.4. Excel как равноправный вход

- [ ] UI загрузки Excel прямо на `/clients` (не только через `/segment`)
- [ ] После загрузки — сразу таблица + автосегментация
- [ ] Сохранить `/segment` для превью и отладки, но основной путь — `/clients`

#### 1.5. Дополнительные источники (контекст, не замена МС)

- [ ] Telegram export: форма загрузки JSON на `/clients` (сейчас только API + автоимпорт с диска)
- [ ] Опционально: обогащение из WhatsApp/TG при включённых ключах (`messenger_enabled`)

> **Не входит в фазу 1:** слияние Excel + МС в одну базу, Postgres, 1С, Instagram.

#### 1.6. Надёжность на Railway

- [ ] Redis для кэша (чтобы рестарт не обнулял базу)
- [ ] Healthcheck: «есть parsed + meta.source + timestamp синка»
- [ ] Логирование ошибок синка / AI в UI (не только в консоль)

### Критерии готовности фазы 1

1. Открываю [prod/clients](https://client-segmentation-deepseek.up.railway.app/clients) — вижу **всех** контрагентов из МойСклад с заказами и позициями
2. AI-поля заполнены **без ручных действий** (или с одной кнопкой «догнать»)
3. Могу загрузить Excel — таблица обновляется, AI отрабатывает
4. Видно, откуда данные и когда последний синк
5. Экспорт XLSX содержит полную обогащённую таблицу

## Фаза 2 — Мини-ChatGPT для аналитики и «горячих кучек»

### Цель

Оператор задаёт вопрос **на естественном языке**, система отвечает на основе **реальных данных** из фазы 1:

> «Какие клиенты самые прибыльные в марте?»  
> «Кто покупал больше 500 тыс. за последние 2 месяца и перестал заказывать?»  
> «Найди клиентов с регулярными заказами на 8 марта для получательниц не из семьи»

Ответ — не галлюцинация, а **вывод с опорой на строки базы**: имена, суммы, периоды, паттерны. Это сырьё для маркетинга и (позже) для фазы 3.

### Принцип работы

```
Вопрос оператора
      ↓
  Query Agent (LLM)
      ↓
  План запроса: фильтры + агрегации + группировки
      ↓
  Выполнение над DataHub (structured query, не «LLM угадывает»)
      ↓
  LLM формулирует человекочитаемый ответ + список клиентов
      ↓
  UI: ответ + кликабельные карточки клиентов + «сохранить сегмент»
```

**Важно:** LLM **интерпретирует** вопрос и **объясняет** результат, но **цифры считаются кодом** (суммы заказов, даты, топ-N).

### Что нужно сделать

#### 2.1. Слой аналитических запросов (Query Layer)

- [ ] Новый сервис `app/services/query_engine.py`:
  - агрегации по клиентам: выручка, кол-во заказов, средний чек, период
  - фильтры по датам, суммам, тегам, группам AI, каналу продаж, статусу
  - топ-N, «отвалившиеся» (нет заказов N дней), сезонность (март, 8 марта, 14 февраля)
- [ ] API: `POST /api/query` — `{ "question": "..." }` → `{ "answer": "...", "clients": [...], "query_plan": {...} }`
- [ ] Покрытие тестами на фикстурах (без вызова LLM для математики)

#### 2.2. Query Agent (LLM)

- [ ] `app/agents/query.py` — `QueryAgent`:
  - вход: вопрос + схема доступных полей + примеры
  - выход: JSON-план (`filters`, `aggregations`, `sort`, `limit`, `time_range`)
  - второй проход: текстовый ответ по **фактическим** результатам query engine
- [ ] Промпт с доменными правилами цветочного CRM (из `segmentation.py`)
- [ ] Guardrails: если данных нет — честно «не нашёл», не выдумывать клиентов

#### 2.3. UI чата на `/clients`

- [ ] Боковая панель или блок «Спросить базу» (как ChatGPT, но по вашим клиентам)
- [ ] История вопросов в сессии
- [ ] В ответе — список клиентов со ссылкой на карточку
- [ ] Кнопка **«Сохранить как сегмент»** → именованный фильтр / тег для кампании

#### 2.4. Готовые «рецепты» (шаблоны вопросов)

Чтобы оператор не гадал, что спрашивать:

- [ ] «Топ-10 по выручке за период»
- [ ] «Клиенты без заказов 30+ дней с LTV > X»
- [ ] «Повторяющиеся заказы в одни и те же даты»
- [ ] «VIP с падением среднего чека»
- [ ] «Новые клиенты с крупным первым заказом»

#### 2.5. «Горячие кучки» — формализация

- [ ] Сущность **Insight** / **HotSegment**: название, описание (AI), список UUID клиентов, дата, автор вопроса
- [ ] Страница или вкладка «Инсайты» — сохранённые ответы фазы 2
- [ ] Экспорт сегмента в XLSX / подготовка к рассылке (мост к фазе 3)

#### 2.6. Качество и доверие

- [ ] В каждом ответе: «На основе N клиентов, M заказов, период …»
- [ ] Раскрытие «как посчитано» (query plan для прозрачности)
- [ ] Лимит контекста: не слать всю базу в LLM — только агрегаты и топ строк

### Примеры вопросов → ожидаемый результат

| Вопрос | Ожидаемый ответ |
|--------|-----------------|
| Какие клиенты самые прибыльные в марте? | Топ-5 с суммами, % от общей выручки марта |
| Кто регулярно заказывает большие букеты? | Список с частотой и средним чеком |
| Кто перестал покупать после февраля? | Клиенты с last_order < 2026-03-01 и LTV > порога |
| Найди похожих на клиента X | По паттерну заказов / AI-группам / суммам |

### Критерии готовности фазы 2

1. Задаю «какие клиенты самые прибыльные в марте» — получаю **корректный топ** с суммами
2. Могу сохранить результат как **сегмент** и увидеть этих клиентов в таблице
3. Ответы **не выдумывают** клиентов — только из `DataHub`
4. 5–10 шаблонных вопросов работают из коробки
5. Оператор за 2 минуты находит «горячую кучку» без Excel и ручных фильтров

## Фаза 3 — Коммуникация и конверсия (будущее)

> Фаза 3 **не начинается**, пока не закрыты фазы 1 и 2. Ниже — ориентир, чтобы понимать, зачем нужны предыдущие этапы.

### Цель

Превратить инсайт фазы 2 в **действие**, которое ув уеличивает конверсию:

*Инсайт:* «Антон и Ганди — 1 млн ₽ за 2 месяца, регулярные заказы в марте»  
*Действие:* персональное предложение (WhatsApp / TG / SMS) с релевантным текстом и таймингом

### Что потребуется (черновик)

- [ ] Модуль кампаний: сегмент → сообщение → канал → отправка
- [ ] AI-генерация текста предложения под сегмент (тон, повод, сумма)
- [ ] Интеграция WhatsApp (Green API) / Telegram Bot — уже частично есть
- [ ] A/B или хотя бы трекинг: отправлено → ответ → заказ
- [ ] Оператор подтверждает отправку (human-in-the-loop на старте)
- [ ] Push тегов обратно в МойСклад (`/moysklad/push-tags`) — уже есть задел

### Критерий успеха всего проекта

**Конверсия:** % клиентов из «горячей кучки», которые сделали заказ после предложения, **выше**, чем у обычной рассылки.

```

1. Прод-синк МС           1. Query engine           Кампании
2. Авто-AI после синка    2. Query Agent            Отправка
3. UI источника/статуса   3. Чат на /clients        Аналитика конверсии
4. Excel на /clients      4. Сохранение сегментов
```

## Риски

| Риск | Митигация |
|------|-----------|
| LLM выдумывает цифры | Считать только кодом; LLM — интерпретация |
| Большая база → медленные запросы | Агрегаты в query engine; индексы; позже Postgres |
| Потеря данных при рестарте Railway | Redis + персистентный кэш |
| Неполный синк МС (лимиты API) | Мониторинг `api_cp_total`, догрузка позиций |
| Оператор не знает, что спрашивать | Шаблоны вопросов + примеры на UI |
